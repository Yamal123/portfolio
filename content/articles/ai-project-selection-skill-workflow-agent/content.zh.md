# AI 项目选型：什么时候用 Skill、Workflow 和 Agent

很多 AI 项目一上来就容易问：“这个要不要做成 Agent？”

但从产品经理视角看，这个问题问早了。真正应该先判断的是：**这件事的不确定性到底在哪里，以及系统需要控制到什么程度。**

Skill、Workflow 和 Agent 不是三个并列的“技术名词”，而是三种不同层级的能力：

- **Skill** 解决“一个确定动作怎么稳定完成”。
- **Workflow** 解决“一条确定流程怎么稳定跑完”。
- **Agent** 解决“面对不确定输入时，系统怎么判断下一步”。

![Skill、Workflow、Agent 的抉择图](/images/blog/ai-project-selection/01-skill-workflow-agent-choice.png)

---

## 一、先看产品问题，再看技术形态

产品经理做选型时，不应该先从“模型能力”出发，而应该先拆业务问题。

可以先问四个问题：

| 判断维度 | 产品侧要看什么 | 技术侧要看什么 |
|---|---|---|
| 目标是否稳定 | 用户是不是每次都要完成同一类动作 | 输入输出能否标准化 |
| 流程是否固定 | 业务步骤能不能提前写成 SOP | 是否适合状态机、DAG 或规则编排 |
| 判断是否复杂 | 是否需要理解语义、补信息、分流 | 是否需要 LLM 做路由或规划 |
| 风险是否可控 | 出错后影响是低成本还是高风险 | 是否需要日志、审计、人工兜底 |

这四个问题决定了系统应该“工具化”“流程化”，还是“智能体化”。

![产品侧与技术侧的选型体检](/images/blog/ai-project-selection/02-product-tech-diagnosis.png)

## 二、什么时候用 Skill

**Skill 适合单步、确定、可复用的动作。**

它在产品上的价值，是把一个高频动作封装成稳定能力；在技术上的形态，通常是一个函数、接口、插件或工具调用。

典型场景包括：

- 查询商品库存
- 抓取网页信息
- 计算佣金率
- 检查违禁词
- 转换文件格式

判断一个能力适不适合做成 Skill，可以看三点：

- 用户目标是不是明确的
- 输入输出是不是结构化的
- 执行过程是不是不需要上下文判断的

如果答案都是“是”，就不必上 Agent。比如“查某个 SKU 是否有库存”，技术上直接走接口更稳，产品上也不需要用户感知任何智能过程。

> Skill 的关键不是聪明，而是稳定、便宜、可复用。

---

## 三、什么时候用 Workflow

**Workflow 适合步骤固定、顺序明确、需要过程控制的业务链路。**

它在产品上的价值，是把业务 SOP 线上化；在技术上的形态，通常是状态机、DAG、审批流、任务流或规则编排。

典型场景包括：

- 内容审核：提交 → 机审 → 人审 → 发布
- 售后处理：识别问题 → 校验凭证 → 计算方案 → 通知用户
- 选品流程：画像分析 → 商品筛选 → 排序 → 话术生成

这里要注意一个产品经理容易踩的坑：**Workflow 里可以有 LLM 节点，但整个流程不应该交给 LLM 自由发挥。**

比如内容审核里，LLM 可以负责“判断文本是否疑似违规”，但“命中高风险后必须进入人审”这种规则，应该由 Workflow 控制。

所以 Workflow 的核心不是“有没有 AI”，而是“谁掌握流程控制权”。

> 流程越需要审计、追踪、回滚和 SLA，越应该优先 Workflow 化。

---

## 四、什么时候用 Agent

**Agent 适合输入模糊、路径不固定、需要动态判断的任务。**

它在产品上的价值，是减少用户的表达成本和决策成本；在技术上的形态，是 LLM 作为决策器，结合上下文、工具、记忆和规则来决定下一步。

典型场景包括：

- 用户只说“帮我选一批适合下周直播的商品”
- 用户需求不完整，需要系统主动补齐信息
- 同一个入口可能要分流到画像分析、趋势分析、商品筛选、话术生成
- 任务过程中需要根据结果继续修正策略

在这类场景里，难点不在执行某个动作，而在“下一步该做什么”。这就是 Agent 的价值。

但 Agent 也不能无限放权。产品经理需要提前定义：

- 它能调用哪些工具
- 它能改哪些状态
- 哪些结果必须交给规则校验
- 哪些场景必须转人工
- 最多允许迭代几轮

否则 Agent 很容易从“智能决策器”变成“不可解释的黑盒”。

> Agent 负责处理不确定性，但不能替代流程治理和风险控制。

---

## 五、真实项目里通常不是三选一

实际落地时，更常见的结构是混合编排：

- **Agent 负责理解需求和动态路由**
- **Workflow 负责主流程推进和状态控制**
- **Skill 负责稳定执行具体动作**

以电商达人选品为例：

- 用户输入一句自然语言需求，Agent 先拆解主题、受众、价格带和数量。
- 如果缺粉丝画像或趋势信息，Agent 决定先补哪类信息。
- Workflow 负责把“画像分析 → 候选召回 → 评分排序 → 话术生成 → 复核”串起来。
- Skill 负责查商品库、查粉丝画像、算分、查违禁词。
- 高客单、医疗保健、不收敛等高风险场景，由 Workflow 强制转人工。

![真实项目里的分层落地](/images/blog/ai-project-selection/03-layered-implementation.png)

这类设计的好处是：Agent 不承担所有执行细节，Workflow 不处理所有语义不确定性，Skill 也不会被迫理解复杂上下文。

---

## 六、给产品经理的选型口诀

可以用一句话收口：

> **简单动作做 Skill，固定链路走 Workflow，动态判断用 Agent，高风险场景加人工兜底。**

再压成一个判断顺序：

1. 如果只是一个稳定动作，先做 Skill。
2. 如果是一条固定业务链路，先做 Workflow。
3. 如果输入模糊、路径不固定，再引入 Agent。
4. 如果涉及钱、合规、医疗、权限、不可逆操作，一定加规则和 HITL。

AI 产品经理的价值，不是把所有东西都包装成 Agent，而是能判断：**哪一层需要智能，哪一层需要确定性，哪一层必须被审计和兜底。**

真正能落地的 AI 系统，往往不是最“智能”的那个，而是最会分层的那个。
